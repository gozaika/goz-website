============================================================
CRITICAL BUSINESS FLOWS
============================================================

------------------------------------------------------------
Consumer claim/payment flow
------------------------------------------------------------

1. Consumer taps Claim.
2. If unauthenticated, redirect to login.
3. Server validates drop visibility, status, audience, pickup window, quantity.
4. Server creates inventory hold by calling api_create_inventory_hold.
5. Server creates Razorpay order/payment intent.
6. Client opens Razorpay Checkout.
7. Client may display pending state, but must not mark order paid.
8. Razorpay webhook verifies payment.
9. Server converts hold to sold/order.
10. order_order status becomes PAID/CONFIRMED/READY_FOR_PICKUP according to schema/state rules.
11. QR nonce and OTP hash are generated server-side.
12. Notification outbox receives confirmation message.
13. Consumer sees QR/OTP confirmation.
14. 30 min before pickup, reminder notification is enqueued.
15. If payment fails or hold expires, reserved inventory is released.

Implement idempotency throughout.

------------------------------------------------------------
Pickup verification flow
------------------------------------------------------------

1. Customer shows QR/OTP.
2. Staff scans QR or enters OTP.
3. Server validates:
   - staff auth
   - staff restaurant access
   - order exists
   - order belongs to restaurant
   - order is pickup-ready
   - pickup window rules
   - nonce/OTP valid
   - not already collected
4. Server transitions order to COLLECTED.
5. Server writes order_pickup_verification_event.
6. Server writes order_status_transition.
7. Server updates relevant drop counters according to DB rules.
8. Staff app shows success.
9. Duplicate attempts show “already collected,” not generic error.
10. Offline verifications are queued and synced safely.

------------------------------------------------------------
Restaurant drop creation flow
------------------------------------------------------------

1. Owner selects or creates bag template.
2. Template has allergens, dietary category, spice level, serves, max holding, min menu value.
3. Publishing a template creates an immutable revision.
4. Drops reference a specific template revision.
5. Owner sets quantity, price, pickup window, type, audience, media.
6. Save as DRAFT.
7. Preview consumer-facing card/detail.
8. Publish to SCHEDULED or ACTIVE depending on publish_at/pickup window.
9. Realtime inventory updates after claims.

------------------------------------------------------------
Admin restaurant onboarding flow
------------------------------------------------------------

1. Partner lead arrives from website/marketing.
2. Admin reviews.
3. Admin creates restaurant entity.
4. Admin records compliance/FSSAI/GST/PAN details.
5. Admin uploads private docs.
6. Admin assigns commission plan.
7. Admin creates owner/team access.
8. Admin activates restaurant.
9. Restaurant portal shows onboarding checklist.

------------------------------------------------------------
DPDP consent flow
------------------------------------------------------------

1. At signup, required operational consent is captured.
2. Optional consents default off.
3. Every toggle creates a privacy_consent_event.
4. Consent is append-only.
5. Notification sends must check latest consent where required.
6. Erasure request creates privacy_erasure_request.
7. Admin reviews.
8. Approved erasure anonymizes profile data while retaining financial/legal records as required.
9. All actions audit logged.
